# 核心概念

## 心智模型

OpenASE 让你像工程经理管理开发团队一样管理一组 AI Agent——只不过你的“开发者”是 AI，你扩展产能的方式不是招更多人，而是写更好的指令。

核心理念：**你是管理者，不是兜底的人。** 当 Agent 交出来的结果不好时，不要自己跳进去补代码，而要改进 Harness（Agent 的执行指令），让*后续类似工单*也一起受益。你就是这样从盯一个 Agent，扩展到并行管理很多 Agent 的。

```
你（管理者）
 │
 │  写更好的指令，而不是更好的代码
 ▼
Harness  ──→  Agent  ──→  代码变更、PR、测试结果
```

## 四个核心概念

使用 OpenASE，只需要先理解四样东西：

```
┌─────────────────────────────────────────────────┐
│                                                 │
│   Ticket  ────→  Workflow  ────→  Agent         │
│   “做什么”       “怎么做”         “谁来做”       │
│                                     │           │
│                                     ▼           │
│                                  Machine        │
│                                  “在哪做”        │
│                                                 │
└─────────────────────────────────────────────────┘
```

| 概念 | 一句话说明 | 类比 |
|------|-----------|------|
| **Ticket（工单）** | 一件要做的事 | 你在 Jira 里分派给开发者的任务 |
| **Workflow（工作流）** | 怎么做这件事的说明 | 给新同事的岗位说明 + 上手文档 |
| **Agent（智能体）** | 干活的 AI | 你团队里的开发者 |
| **Machine（机器）** | 干活的环境 | 开发者的笔记本 / 开发服务器 |

这就是核心模型。其他能力——Skill、Scheduled Job、Activity、Project AI——都是建立在这四个概念之上的。

### Ticket —— “做什么”

一个工单描述一项工作：修一个 bug、做一个功能、写一组测试。写法和你给人类开发者写任务一样——要有清晰背景、期望结果、相关代码位置。

工单本身不负责说*怎么做*，那是 Workflow 的职责。

### Workflow —— “怎么做”

Workflow 是工单和 Agent 之间的桥梁，它包含：

- **Harness**：一个 Markdown 文档，告诉 Agent 扮演什么角色、采用什么方式、验收标准是什么
- **状态绑定**：哪些工单状态会触发工作流领取任务，以及完成后可以流转到哪些状态
- **配置项**：并发上限、超时、重试行为

Harness 是最值得你投入的地方。模糊的 Harness（“请完成这个任务”）会产出模糊结果。具体的 Harness（“你是后端工程师，先读失败测试，再定位根因、修复问题、跑测试、开 PR”）会得到更稳定的结果。

### Agent —— “谁来做”

Agent 是注册好的 AI 提供商（Claude Code、Codex、Gemini CLI），负责真正执行工作。注册一次、绑定到工作流后，它就会自动领取匹配的工单。

你可以**并行运行多个 Agent**，每个绑定不同工作流——一个写代码、一个跑测试、一个做评审。这才是杠杆：不是让一个 AI 串行做完所有事情，而是让一个小团队同时推进。

### Machine —— “在哪做”

Machine 是执行环境——一台服务器、容器，或你的本地机器，Agent 会在上面跑命令、读代码、写文件。至少先注册一台 Machine，之后你才能让 Agent 自动执行工单。

## 进阶扩展：Skill

当四个核心概念跑通之后，**Skill** 让你把可复用能力抽出来：

| 没有 Skill | 有 Skill |
|-----------|----------|
| 每个 Harness 都复制粘贴相同的 Git 流程说明 | 写一个“Git PR” Skill，给任意工作流复用 |
| 每个工作流都重新拼部署步骤 | 抽一个“Deploy” Skill，在多个部署工作流里共享 |

Skill 不是必需的，但随着工作流数量增长，它会越来越有价值。

## Project AI：你的规划搭档

你不需要从零开始手写每一个工单。**[Project AI](./project-ai.md)** 是项目侧边栏里的持久化同步助手——你可以像和同事聊天一样，跟它澄清需求、在隔离工作区里检查代码、起草甚至直接创建工单。

典型流程：

```
模糊的想法  ──→  和 Project AI 对话  ──→  结构清晰的工单
                                           │
                                           ▼
                                     Ticket Agent 执行
```

Project AI 最擅长的是规划和探索：分析需求、检查代码、看 diff、帮你写 Ticket / Harness / Skill。Ticket Agent 则负责排队的异步执行流程。至于 Project AI 能不能直接回写项目状态（例如 ticket、workflow、skill、update），要看项目里 **Project AI Platform** 配置给它哪些权限。

## 核心循环：通过 Harness 管理

这是 OpenASE 最重要的理念：

```
 ┌──────────────────────────────────────────────────────┐
 │                                                      │
 │    创建工单（手动或通过 Project AI）                    │
 │        │                                             │
 │        ▼                                             │
 │    Agent 自动执行                                     │
 │        │                                             │
 │        ├── 结果好？  ──→  继续下一个工单                │
 │        │                                             │
 │        └── 结果差？  ──→  改进 Harness                 │
 │                                │                     │
 │                                │                     │
 │               未来所有工单都受益                        │
 │                                                      │
 └──────────────────────────────────────────────────────┘
```

**不要**每次都手动修 Agent 的产出然后继续往前走。那只修好了一个工单，却浪费了一次系统性学习机会。正确做法是：

1. 找出哪里出了问题
2. 改 Harness，把约束、步骤、验收标准写得更清楚
3. 让 Agent 重新执行，或者让下一个类似工单自动做对

这就是你从“盯着一个 AI 干活”升级成“管理一支 AI 队伍”的方式。Harness 是你的杠杆——每次改进都会在未来执行里持续放大收益。

### 要避免的反模式

| 反模式 | 为什么有害 | 应该怎么做 |
|--------|----------|-----------|
| 每次手动修 Agent 的产出 | 你会变成瓶颈，Agent 也学不到东西 | 改进 Harness，让它下次自己做对 |
| 一个巨大的 Workflow 包办一切 | 难调试，也难并行 | 拆成更聚焦的工作流：编码、测试、评审、部署 |
| 跳过 Harness（“直接做工单”） | Agent 只能猜你的标准，容易猜错 | 写清楚 Harness，每个工单都会回本 |
| 不设并发和超时 | Agent 可能堆积或卡住太久 | 给每个工作流设置 `MaxConcurrent` 和 `TimeoutMinutes` |

## 实战演练：从零到并行 AI 团队

一个具体例子——你有一个 Python 后端项目，希望 AI Agent 并行处理 bug 修复和测试。

### 1. 一次性配置（基础设施层）

- **Settings**：创建一组状态，例如 `To Do → In Progress → In Review → Done`，并关联 GitHub 仓库
- **Machines**：注册你的开发服务器（或使用本地机器）
- **Agents**：注册两个 Agent——例如 “Claude-Coder” 和 “Claude-Tester”

### 2. 定义怎么干活（模板层）

- **Workflow “Bug Fix”**：绑定 Claude-Coder，编写 Harness：
  ```markdown
  # 角色
  你是一个负责 Python 后端项目的工程师。

  # 任务
  修复这个工单描述的 bug：
  - 标题：{{ ticket.title }}
  - 描述：{{ ticket.description }}

  # 步骤
  1. 阅读相关代码，复现问题
  2. 定位根因
  3. 编写修复代码和回归测试
  4. 运行完整测试套件
  5. 提交 PR 并写清描述

  # 约束
  - 不要改变公开 API 签名，除非在 PR 中特别说明
  - 所有测试必须通过才能提交 PR
  ```
- **Workflow “Test Suite”**：绑定 Claude-Tester，Harness 专注于写测试和跑测试
- **Skills**：给两个工作流都绑定一个共享的 “Git PR” Skill，统一分支与提交流程

### 3. 日常使用（执行层）

- 创建工单——手动写，或者告诉 [Project AI](./project-ai.md) 你的需求，让它帮你起草或直接创建
- 工单进入 pickup 状态后，对应工作流的 Agent 会**自动**执行
- 两个 Agent 可以**并行**处理各自负责的工单
- 在 **Activity** 里观察进度，PR 落地后再审查

### 4. 持续改进

- Claude-Coder 总忘记跑 `ruff check`？把它补进 Harness 的 Constraints。
- Claude-Tester 写的测试总漏边界条件？就在 Harness 里把边界场景写明白。
- 每一次 Harness 改进都会惠及未来工单——你是在管理一个系统，而不是亲自完成每一项执行细节。

## 概念全景图

完整关系可以这样理解：

```
Settings ── 定义状态、关联仓库
   │
   ├── Machine ── 注册执行环境
   │
   ├── Agent ── 注册 AI 提供商
   │
   ├── Skill ── 创建可复用能力包（可选）
   │      │
   │      ▼
   ├── Workflow ── 绑定 Agent + Harness + Skill + 状态触发
   │      │
   │      ▼
   ├── Ticket ── 创建工单，关联 Workflow → Agent 自动执行
   │      │
   │      ▼
   ├── Scheduled Job ── 定时自动创建工单（可选）
   │
   ├── Activity ── 实时执行日志
   │
   ├── Updates ── 手动发布的项目动态
   │
   └── Project AI ── 持久化助手，用于规划、检查工作区、起草工单
```
