# Project AI

## 这是什么？

Project AI 是项目侧边栏里的**持久化、交互式 AI 助手**。Ticket Agent 负责按工单异步执行任务，而 Project AI 更像是和你结对工作的资深工程师：你直接和它对话，它也有自己独立的工作区。

可以这样理解：

| | Ticket Agent | Project AI |
|---|---|---|
| **模式** | 异步——由工单状态和工作流触发 | 同步——你实时发起对话 |
| **触发** | 工单进入 pickup 状态后执行 | 你打开会话并发送消息 |
| **范围** | 一次处理一个工单执行 | 项目级上下文 + 当前会话工作区 |
| **最擅长** | 执行明确的任务 | 探索、规划、起草、检查、编辑 |

有一个重要边界：Project AI 在项目里到底能改什么，取决于该项目的 **Project AI Platform** 权限配置。默认会给完整的 project conversation scope 集合，但管理员可以把它收紧。

## 什么时候用 Project AI

- **需求还不清楚**——先把想法聊明白，再决定如何拆工单
- **需要起草或创建工单**——用自然语言描述需求，把它整理成结构化工单
- **想编辑 Harness 或 Skill**——在交互里迭代，比手写文档更快
- **需要检查项目状态**——查看工单上下文、Agent 运行、活动日志、仓库 diff、机器健康
- **想直接在隔离工作区里动手**——浏览文件、搜索代码、切换分支、做小范围编辑

## 能力一览

### 读取项目状态

- 工单详情、依赖关系、最近活动、Hook 历史
- Workflow 元数据和 Harness 内容
- Skill 包内容和工作流绑定关系
- Machine 的健康状态、主机信息和连接摘要
- 当前会话工作区的分支状态与 diff 摘要

### 创建和管理工作

- 在项目授予 ticket 写权限时，**起草或直接创建工单**
- 更新工单和评论，边讨论边收敛范围
- 在启用项目级写权限时，读取或发布项目 Updates

### 编写自动化

- 查看和编辑 Workflow 与 Harness
- 在权限允许时创建、导入、更新、启用、停用、绑定 Skill
- 在把 Harness 交回给 Ticket Agent 前先做校验

### 在会话工作区里工作

- 浏览仓库树、搜索路径、预览文件
- 在当前会话的隔离工作区中编辑并保存文件
- 查看该工作区的 git diff 摘要
- 在仓库干净时切换工作区 repo 的分支
- 打开终端会话，做更深入的检查或一次性命令执行

### 操作项目状态

- 在相应 scope 开启时更新项目元数据或仓库绑定
- 在 ticket focus 下检查 Agent 当前状态
- 当某个 ticket 处于活动运行中时，可在 ticket focus 下中断当前 Ticket Agent

## 隔离工作区

每个 Project AI 对话标签页都有自己**独立的隔离工作区**，这意味着：

- 多个标签页可以同时处理不同任务
- 每个标签页都有自己的分支和文件状态
- 工作区 diff 只属于当前会话，审查变更更可控

这和并行 Agent 运行使用的是同一套隔离思路，可以避免不同会话互相踩工作区。

## 上下文感知聚焦

Project AI 会从受支持的界面自动携带结构化 focus。当前一共有 **5 种** focus：

| 聚焦模式 | 何时激活 | 会注入什么 |
|---------|---------|-----------|
| **Workflow** | 查看工作流编辑器时 | 工作流元数据、Harness 路径、选中区域、未保存草稿提示 |
| **Skill** | 查看 Skill 编辑器时 | Skill 名称、当前文件、绑定工作流、未保存草稿提示 |
| **Ticket** | 查看工单时 | 工单详情、依赖、最近活动、当前运行、分配的 Agent |
| **Machine** | 查看机器时 | 主机、状态、健康摘要 |
| **Workspace File** | 在 Project AI 工作区浏览器/编辑器里工作时 | repo 路径、文件路径、选区上下文、草稿状态、working set |

大多数情况下，你不需要先手动复制粘贴上下文——focus capsule 已经把最相关的信息带进来了。

## 最佳实践

### 把 Project AI 当作规划层

更高效的模式通常是：

1. **从一个模糊想法开始**——先和 Project AI 讨论
2. **收敛方案**——让它查看代码库、比较路线、指出约束
3. **起草或创建工单**——把结果沉淀成结构化工作项
4. **交给 Ticket Agent 执行**——后续由异步流水线接管

Project AI 更适合探索和决策；Ticket Agent 更适合可重复的排队执行。

### 用 Project AI 改进 Harness

当 Ticket Agent 结果不理想时，打开 Project AI：

1. 回顾工单上下文、运行状态和 diff
2. 找出 Harness 缺少的约束或步骤
3. 编辑 Harness 或相关 Skill，把要求写清楚
4. 让下一次工单执行直接受益

### 用 workspace focus 做精确修改

如果你已经在 Project AI 的工作区浏览器里，先聚焦到具体文件或选区再提问。这样助手能拿到更准确的 patch 目标、上下文和草稿状态。

## 小贴士

- Project AI 对话会持久化保存，可以随时续聊
- 不同标签页可以使用不同的 AI 提供商（如 Claude Code、Codex、Gemini CLI）
- Project AI 是否能回写 OpenASE，取决于项目 **Project AI Platform** 里允许的 scopes
- 需要并行调查多个方向时，开多个标签页比复用一个工作区更稳妥
